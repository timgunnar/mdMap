# mdMap — Markdown 文档地图

mdMap 是一个零依赖 CLI，将你的 markdown 文件索引为结构化 JSON 地图，把约束从文档内部提取到表面。

> **Markdown 文档索引与查询引擎。告诉它你在做什么，精准找到文档，LLM 上下文不再白白烧掉。**

## 为什么要有地图

### 无地图时代：Agent 实际是怎么找文档的

一个项目里有几百篇 `.md`。Agent 要找发布指南。现实是这样的：

**第一步 — 用 grep 搜内容。** Agent 有 grep 工具。不是搜文件名，而是搜文件**内容**。跑 `grep -rl "发布" docs/`，遍历每个文件的每一行，找出包含"发布"这个词的文件：

```
publish_checklist.md           ← 完全正确
auth_v3.md                     ← 某一行写着 "发布 token..."
architecture_overview.md       ← 提了一句 "发布之后..."
meeting_notes_2026-05.md       ← "讨论了发布流程时间线"
deploy_guide.md                ← "与发布流程类似"
release_guide.md               ← 真正的发布指南，但全文不说"发布"——它说的全是"release"
```

grep 返回 20 条路径。有几条是完全匹配，有几条只是顺便提到这个词，有一条关键文档——`release_guide.md`——根本不可见，因为用的是不同的词。这就是**词汇错配**：Agent 搜"发布"，文档从头到尾说的是"release"。

**第二步 — 逐个打开文件来验证。** Agent 没法从 grep 返回来判断这 20 个文件哪个真的相关。grep 只返回路径（最多附带断章取义的匹配行）。Agent 必须逐个打开文件来验证：

```
Read(auth_v3.md)              → 认证文档，只写了一句"发布 token" → 关掉。-2K tokens。
Read(architecture_overview.md) → 架构文档，顺便提了一句 → 关掉。-1.5K tokens。
Read(deploy_guide.md)          → 部署指南，"与发布流程类似" → 关掉。-2K tokens。
Read(publish_checklist.md)     → 对了！这是发布检查清单 → 读完。-3K tokens。
```

开了三扇错门，找对一扇。5.5K tokens 花在了不包含所需内容的文件上。

**第三步 — 追踪隐藏的交叉引用。** 在 `publish_checklist.md` 里，Agent 读到：*"发布前先检查 security_policy.md 的安全策略。"* 又有：*"如果发布到 GitHub，参考 release_guide.md。"* Agent 继续打开这些文件。又是 4K tokens。这些交叉引用困在文档内部——不打开就不可见。

**第四步 — 下次重来。** Agent B 明天开始类似的任务。grep "发布" → 20 条路径 → 打开 4 个文件 → 烧 10K tokens。Agent A 今天花的力气不传递。每个 Agent、每次 session，都在重复发明同样的导航路径。

**算一笔账。** 一次典型的文档导航任务：

| 阶段 | token 消耗 | Agent 真正需要的 |
|------|----------|---------------|
| grep | ~200 | 路径——只是线索 |
| 打开错误的文件 | ~5,500 | 毫无用处 |
| 追踪交叉引用 | ~4,000 | 有用，但靠运气发现 |
| 读正确的文件 | ~3,000 | 真正的目标 |
| **合计** | **~12,700** | **只有 3,000 是有效内容** |

~76% 的上下文烧在导航上。Agent 不是在猜文件名——它确实在阅读。它只是反复阅读了错误的东西。

### 革命

mdMap 用一次查询取代整个流程：

```
mdmap find --search "发布"
# [checklist]  publish_checklist.md  — 发布工具到 GitHub 的分步指南
# [rule]       security_policy.md    — 所有发布操作的安全要求
# [checklist]  release_guide.md      — 完整发布流程（GitHub releases、changelog、npm）
```

一条命令。三条结果。Agent 直接读到摘要——`publish_checklist.md` 就是它要的。打开那个文件。零扇错门。

但注意第三条结果：`release_guide.md`。grep 搜"发布"永远找不到它——"发布"这个词在这篇文档里没出现过。mdMap 找到它，是因为之前的 Agent 读过这篇文档，理解了它讲的是 **release**，然后在 triggers 字段里标注了"发布工具"。Agent 弥合了搜索词汇和文档用词之间的鸿沟。

以下是 mdMap 赋予每条结果的、grep 提供不了的信息：

| grep 返回 | mdMap 返回 |
|----------|----------|
| 裸文件路径 | 路径 + **类型标签**（`[rule]`、`[checklist]`）——一眼知道文档角色 |
| 断章取义的匹配行 | **一句话摘要**——不打开就知道文档讲什么 |
| 无状态信息 | **状态标签**（`[deprecated]`、`[draft]`、`[archived]`）——知道该不该读 |
| 无关系数据 | **links**——地图记录了文档之间的指向关系，不进门就全看见 |
| 无知识积累 | **累积索引**——Agent A 的标注，Agent B、C、D 永远受益 |

而且每有 Agent 走过就变得更好。Agent A 填了 publish_checklist.md 的 triggers。Agent B 加了指向 security_policy.md 的链接。Agent C 发现 auth_v2.md 已经 deprecated 并标记了它。索引有机增长——不在一次昂贵的全量遍历中完成，而是在 Agent 实际工作中一篇一篇地积累。

地形——你的 `.md` 文件——纹丝不动。Go CLI（`sync`、`find`、`validate`、`changed`）制作卡片、查询卡片、检查卡片。Agent 标注卡片。你还可以继续 grep。你还可以继续直接读文件。但你再也不需要在导航上烧掉 76% 的上下文窗口了。

## 规则

**不知道要打开哪个文档时，先试 mdMap。** 如果返回结果，打开第一条。如果返回空（索引是空白——fresh sync 之后常见），扫目录，按文件名挑最像的那个。

## 地图是怎么画出来的

有人跑过 `mdmap sync`——制图工具扫描了每一条街，画出一张空白地图。每栋楼以空白条目的形式出现：它在真实地形上存在，但地图对它的描述为空。

然后 Agent 开始走路。每有人走进一栋楼，就在地图上标注：这栋楼应该挂什么颜色的旗子、贴什么状态牌、门口有什么提示标语、指向哪些其他建筑。地图一条街一条街地填满。

地图存在 `mdMap.json` 里。图例（每个字段什么意思、哪些旗色和牌类有效）存在 `SCHEMA.md` 里。永远不要读完整 `mdMap.json`——始终通过 CLI 查询。

## 地图的隐藏力量：标志与连接

两项功能让地图比光走地形更有用：

**标志（triggers, maintains, retires）。** 当有人走进一栋楼，他们读墙上的提示——*"发布东西前来这里"*、*"认证改了来复核这里"*。他们把那些提示抄到地图上。现在你可以搜"发布"就找到那栋楼，连门都不用推，连隔壁都不用去。每一块本被困在文档里的指示牌，都被索引到了表面。

**连接（links）。** 楼里有指向其他楼的方向指引——*"发布前先看 security_policy.md"*、*"已被 auth_v4.md 替代"*。进去过的人把那些连接记在了地图上。现在你不用亲自走，也能追踪整个路网——看哪些楼指向哪里、发现断头路、检测环路。

```bash
# 门口的提示牌——不用进门就能搜
mdmap find --trigger "发布"       # "哪些楼门口写着'发布东西的来这里'？"
mdmap find --maintains "认证改了"  # "认证变更后哪些楼需要复核？"
mdmap find --retires "CLI 废弃"   # "哪些楼现在已经过期了？"

# 地图自己追踪连接
mdmap validate  # 检查：有没有路牌指向已拆除的楼？
                #       有没有楼互相指成环？
                #       有没有楼根本没在地图上？
```

## 如何查看地图

`mdmap find` 就是查地图。每个 flag 是一个筛选维度。

```
找到所有挂旗子=rule、状态牌=active、名字或描述里提到"发布"的建筑。

  --type rule     建筑类别（rule = 红旗，checklist = 清单……）
  --status active  建筑状态（active、deprecated、draft、archived）
  --search "发布"  建筑名称或描述里的文字
```

底层引擎就是 SQL：精确字段用 `=` 过滤，语义字段用 `LIKE` 子串匹配。

**地图最多返回 2-5 栋建筑。** 你对 2-5 条一句话摘要的理解判断，远胜任何算法。第一条不对就看第二条。

```bash
# 要发布 project-a → 大概是规则类文档
mdmap find --type rule --search "project a"
# [rule]  project_a_publish.md  — 发布 project-a 的规则：GitHub releases
# [rule]  project_b_publish.md  — 发布 project-b 的规则：npm publish

# 第一条摘要对上了 → 打开。结束。
```

**为什么 trigger 和 search 是分开的：**

| flag | 查什么 | 比喻 |
|------|-------|------|
| `--search` | title, summary, positioning | 建筑的名字和一行简介 |
| `--trigger` | `triggers` 列表 | 从门口抄下来的提示牌（"发布东西的来这"） |
| `--maintains` | `maintains` 列表 | 关于维护的提示牌（"认证变了来这"） |
| `--retires` | `retires` 列表 | 关于退役的提示牌（"CLI 不做了来这"） |
| `--type` | document type | 建筑上挂的旗子颜色 |
| `--status` | document status | 建筑上贴的状态牌 |
| `--tag` | tags | 建筑上的自由标签 |

search 查的是建筑本身是什么。trigger/maintains/retires 查的是门口牌子写了什么。不同维度，同一张地图。

## 建筑分类

每栋建筑在地图上挂着一面彩色旗子。两种旗色是必须的：

| 旗子 | 含义 | 规矩 |
|------|------|------|
| **`rule`**（红旗） | 这栋楼约束了工作**必须怎么做** | **必须进。** 忽略它等于不遵守项目规范。例：编码规范、安全策略、架构原则 |
| **`resource`**（蓝旗） | 独立长文本，和其他楼没有连接关系 | **好奇可以进。** 它不约束你的行为。例：小说章节、世界观设定文档、历史参考笔记 |

其他旗色由项目自然产生：`checklist`、`architecture`、`guide`、`api_spec`、`meeting_notes` 等。标注新楼时，沿用已有的旗色保持统一。

地图把旗子显示在行首：

```bash
mdmap find --search "auth"
# [rule]        security_policy.md    — API 鉴权必须遵守的安全策略
# [checklist]   auth_migration.md     — 认证系统迁移检查清单
# [resource]    auth_history.md       — OAuth 协议演进历史文档
```

**红旗（`rule`）= 先打开。蓝旗（`resource`）= 需要参考才打开。**

## 建筑状态

每栋建筑贴着一块状态牌。四种状态牌是必须的：

| 牌子 | 含义 | 怎么做 |
|------|------|--------|
| **`active`** | 正常运营，当前版本，权威 | **进去。这就是你要读的那栋。** |
| **`deprecated`** | 已报废——被替代或不再适用 | **不要把它当作主要参考。** 地图显示 `[deprecated]` 警告。进去只是为了找到替代者。 |
| **`draft`** | 在建——内容可能变化 | **进去看看方向可以。** 不能作为最终依据。 |
| **`archived`** | 历史遗迹——仅作保存 | **除非明确要求，不要进去。** 不属于活跃知识网络。 |

`active` 不显示（默认状态，不制造噪音）。非 active 状态会标出：

```bash
mdmap find --search "auth migration"
# [checklist]   auth_migration_v3.md   — 当前认证迁移检查清单（v3）
# [checklist]  [deprecated]  auth_migration_v2.md  — 旧版检查清单（v2）
# [guide]      [draft]  auth_migration_v4.md  — 新版指南（起草中）
```

## 双轨制架构

你是一个拿着地图、走在真实地形上的人。

| 元素 | 它是什么 | 在 mdMap 中 |
|------|---------|------------|
| 地形 | 真实地理——你脚下的街道和建筑 | 磁盘上的 `.md` 文件 |
| 地图 | 地形的平面表示——街道名、标注、笔记 | `mdMap.json` |
| 制图工具 | 制作、对比、检查地图的仪器 | `mdmap sync / changed / validate / find` |
| 你 | 走路、阅读、判断、标注的人 | Agent |

**工具不走地形。** 它们扫描目录生成地图，对比地图版本和磁盘，检查地图内部一致性，查找位置。零 LLM。纯结构。

**地图也不走地形。** 它是一张平面的纸——记录了上次你来这里时的标注。可能准确，可能过时，可能空白，可能错误。

**你走地形。** 你读真正的 .md 文件。你把看到的东西和地图上说的做对比。你判断地图需不需要更新。

### 决策一：要不要翻地图？

```
┌─────────────────────────────────┐
│ 你已经知道要打开哪个文件吗？     │
└────────────┬────────────────────┘
             │
     ┌───────┴───────┐
     │ 知道           │ 不知道
     ▼               ▼
  直接打开。      ┌─────────────────────────┐
                  │ 地图上这片区域可能有     │
                  │ 标注吗？                │
                  │ （你或别人之前走过       │
                  │  这些街道吗？）          │
                  └──────────┬──────────────┘
                             │
                     ┌───────┴───────┐
                     │ 大概率有       │ 大概率空白
                     ▼               ▼
               查地图：          自由探索：
               mdmap find       扫目录，
               --search         靠文件名判断
```

**不看地图直接走：**
- 你知道确切路径，去过。
- 项目刚 `mdmap sync`，全是空的。
- 你用 `find` 查了，没结果——地图没覆盖这片。

**查地图：**
- 你不知道该打开哪个文件。
- 你要找到某类活动涉及的全部文档（`--type rule`）。
- 你想知道一篇文档是 active 还是 deprecated。
- 多个 Agent 在这里工作过——别人可能标注了你没走过的街道。

### 决策二：读完一篇文档，要不要更新地图？

你因任何原因读了一篇文档——任务、好奇、`find` 的结果。读完之后，问：

```
┌─────────────────────────────────────┐
│ 这篇文档在 mdMap.json 里吗？         │
└────────────┬────────────────────────┘
             │
     ┌───────┴───────┐
     │ 不在           │ 在
     ▼               ▼
  更新地图。      ┌─────────────────────────────────────┐
  这是新地点。    │ 你刚读到的内容和地图上说的            │
                  │ 一致吗？                            │
                  │                                     │
                  │ 对比：title, type, summary,          │
                  │ status, triggers, maintains,         │
                  │ retires, links                       │
                  └──────────────┬──────────────────────┘
                                 │
                         ┌───────┴───────┐
                         │ 一致           │ 不一致
                         ▼               ▼
                    什么都不做。       更新地图。
                    地图已经是        修正错误。
                    准确的。          地形变了，地
                                     图没有跟上。

                    例外：            例外：
                    文档在 map        文档在 map
                    中是空的         中标记 [active]
                    （sync 后），但你在文档里
                    即使觉得内容     发现了 "已被
                    很简单也要       ... 替代" →
                    填上。           改成 [deprecated]。
```

**需要更新：** 地图里根本没这篇文档，地图里字段是空的（sync 之后），地图的 type/summary/status/triggers 和你刚读到的不一致，文档引用了地图里没记录的其他文档，文档说已被替代但地图标着 active。

**不需要更新：** 地图已经准确描述了这篇文档。你读了，确认了地图上的一切，没有发现任何与地图矛盾的地方。不是每次经过都要在地图上写一笔——只有发现差异或填补空白时才写。

### 决策三：地图和地形打架了怎么办？

地图不光是信息不全——有时干脆是错的。

| 地图上写的 | 但地形显示 | 怎么回事 | 怎么做 |
|-----------|----------|---------|-------|
| "auth_v3.md"（status: active） | 文档里写着 "已被 auth_v4.md 替代" | 有人更新了文档但没更新地图 | auth_v3 → deprecated，填 retires |
| "rules.md"（type: checklist） | 文档里全是约束和规范 | type 分类错了 | 改成 type → rule |
| trigger: "发布" | 文档实际讲的是部署，不是发布 | trigger 有误导 | 换成准确的 trigger 关键词 |
| links → "old_design.md" | old_design.md 在磁盘上不存在 | 链接目标已被删 | 删除或更新这个 link |
| 文档在地图里，但磁盘上没这个文件 | — | 文件被删/移动但地图没更新 | 跑 `mdmap sync` 同步，或 `mdmap validate` 检测 |

**这些差异是你走出来的。** 地图不会自己告诉你它错了——只有地形能。你每次打开一篇文档，都在用地形验证地图。

**工具也能帮你发现一些：**
- `mdmap validate` 发现：磁盘有但地图没有（孤儿）、地图有但磁盘没有、链接指向 deprecated 文档、链接成环。
- `mdmap changed` 发现：磁盘和上次 `sync` 相比，新增或删除了哪些文件。

### 决策四：地图是怎么越变越好用的？

每有人走一条没走过的街道，地图就变得更好一点。

```
第一天：mdmap sync → 200 条空白条目
        Agent A 走了 5 条街，标注了它们
        地图：5 条有标注，195 条空白

第二天：Agent B 走了 3 条 Agent A 走过的街 → 确认标注仍准确
        Agent B 走了 7 条新街 → 标注了它们
        地图：12 条有标注，188 条空白

第三天：Agent A 走了一条自己第一天标注过的街
        → 文档从第一天后更新过 → summary 对不上了
        → Agent A 更新地图
        地图：12 条有标注（1 条更新），188 条空白

第 N 天：80 条街有标注。find --search 真的好用了。
         120 条还是空白——但空白是因为还没人需要它们。
```

**别一次性批量标注整个地图。** 只标注你真的走过的街道。一条没人需要的街道，空白着完全没问题。等终于有人需要它了，走过去顺便标注就行。

**多个 Agent 共享同一张地图。** Agent A 昨天标注的，Agent B 今天受益。Agent B 发现是错的，Agent A 明天就不会踩坑。

### 总结：所有场景

| 场景 | 怎么做 |
|------|-------|
| 地图空白，你要找文档 | 自己扫目录。靠文件名判断。读最像的。在地图上标注它。 |
| 地图有标注，你要找文档 | `mdmap find --search "..." --type rule`。读第一条结果。 |
| find 返回空（地图没覆盖） | 自己扫目录。靠文件名判断。读完标注地图。 |
| find 返回了结果，你读了文档 | 把文档内容和地图元数据做对比。一致 → 结束。不一致 → 更新地图。 |
| 你已经知道文件路径 | 直接打开。不需要看地图。读完之后仍然检查：地图准确描述了这里吗？ |
| 你读了文档，它引用了其他文档 | 更新地图的 `links` 字段。以后的 Agent 搜这篇文档就能发现那些关联，不用打开文件。 |
| 你新建了一个 .md 文件 | 在 mdMap.json 添加条目，填好语义字段。不要等 `sync` 捡到一个空白条目。 |
| `mdmap validate` 报告孤儿 | 磁盘有新文件但地图里没有。读它们，添加条目到 mdMap.json。 |
| `mdmap validate` 报告断裂链接 | 某文档链接了不存在的东西。检查地形——目标被删了？移动了？修正或删除链接。 |
| `mdmap validate` 报告陈旧链接 | 某文档链接了 deprecated/archived 文档。要不要把链接更新到替代版本？ |
| `mdmap validate` 报告环路 | 文档互相链接成环。通常是错误——修正链接结构。 |
| `mdmap changed` 报告新文件 | 有人往磁盘加了 .md 文件。读它们，添加条目到 mdMap.json。 |
| `mdmap changed` 报告删除文件 | 文件从磁盘上消失了。跑 `mdmap sync` 清理地图，或手动删除条目。 |
| 你怀疑地图过时了 | 跑 `mdmap changed`。它会精确告诉你磁盘和上次 `sync` 有什么不同。 |
| 地图说 [deprecated]，你要找当前版本 | 读 deprecated 文档找到谁替代了它。沿着链接走。如果地图缺了这条链接，补上。 |
| 你读的文档在地图里信息不全 | 把缺的字段填上。哪怕只加一个 summary 也比空白强。 |

## 制图工具箱

```bash
# 画地图 — 扫描每条街道，生成空白条目
mdmap sync ./docs

# 查地图 — 按旗色、状态牌、门口提示、标签查建筑
mdmap find --type rule --search "发布"
mdmap find --trigger "发布 CLI 工具"
mdmap find --type checklist --tag "发布"
mdmap find docs/architecture/auth_v3.md      # 按地址查
mdmap find --trigger "认证" --json            # 机器可读输出

# 健康检查 — 地图和地形一致吗？
mdmap validate          # 孤儿、断裂链接、环路、陈旧引用
mdmap validate --strict # CI 门禁（warning 也报错）

# 地形变化 — 上次画地图以来，地形变了什么？
mdmap changed           # 新建筑、已拆除建筑
```
